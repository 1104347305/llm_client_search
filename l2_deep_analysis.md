# L2规则测试结果深度分析

## 测试结果
- 完全匹配: 1/48 (2.1%)
- 部分匹配: 2/48 (4.2%)
- 未匹配: 45/48 (93.8%)

## 核心问题：预处理破坏了匹配模式

### 问题1: value_mappings预处理后，规则pattern无法匹配

**案例1**: "所有万能险客户名单"
- 预处理后: "所有万能客户名单"
- 问题: "万能险"→"万能"，但规则pattern期望匹配"万能险"或"万能保险"
- 原因: value_mappings将"万能险"替换为"万能"，但规则的patterns_template是`{enum}(?:产品|保险)?`，期望enum="万能"后面可选"产品|保险"，但实际文本中"险"已被删除

**案例2**: "给我看看意外险名单"
- 预处理后: "给我看看意外伤害保险名单"
- 问题: "意外险"→"意外伤害保险"，但规则无法匹配"意外伤害保险名单"
- 原因: patterns_template是`{enum}(?:产品|保险)?(?:的客户|客户)?`，不包含"名单"结尾

**案例3**: "A1有哪些"
- 预处理后: "A1类客户有哪些"
- 问题: "A1"→"A1类客户"，但规则无法匹配"A1类客户有哪些"
- 原因: patterns_template不包含"有哪些"模式

**案例4**: "男性年交保费30万以上"
- 预处理后: "男年交保费30万以上"
- 问题: "男性"→"男"，但规则期望"男性"或"男"后面有特定模式
- 原因: value_mappings将"男性"→"男"，但规则pattern可能期望完整的"男性"

### 问题2: 规则pattern覆盖不足

**缺少的常见模式**:
1. "名单" - "意外险名单"、"A类低温名单"
2. "有哪些" - "A1有哪些"、"找40岁左右有哪些人"
3. "的人" - "查50岁以上的人"
4. "及以上" - "本科及以上"、"中温及以上"
5. "做X的客户" - "做老师的客户"
6. "购买过X" - "购买过盛世金越"、"买过年金险"
7. "但" - "有娃但没配置"、"已婚但未配置"
8. "且" - "购买了车险且客户价值为A1"

### 问题3: 组合规则未触发

**案例**: "30-40岁的已婚有娃姓女性客户"
- 预处理后: "30-40岁的已婚有子女姓女客户"
- 问题: 包含年龄范围、婚姻、家庭成员、性别，但完全未匹配
- 原因: 可能没有对应的composite_rule，或者"姓女"破坏了模式

### 问题4: 预处理过度简化

**案例**: "女性客户、30多岁、买过年金险的"
- 预处理后: "女客户、30多岁、买过年金保险的"
- 问题: "女性"→"女"，"年金险"→"年金保险"
- 影响: 简化后的文本可能不符合规则pattern

## 根本原因分析

### 设计矛盾
1. **value_mappings的目的**: 将别名归一化为标准值，方便规则匹配
2. **实际效果**: 预处理后的文本反而更难匹配，因为：
   - 规则pattern是基于原始查询设计的（如"意外险"、"万能险"）
   - 预处理将其替换为标准值（如"意外伤害保险"、"万能"）
   - 但规则pattern没有相应调整

### 例子说明
- 用户输入: "意外险名单"
- value_mappings: "意外险"→"意外伤害保险"
- 预处理后: "意外伤害保险名单"
- 规则pattern: `{SEARCH}{enum}(?:产品|保险)?(?:的客户|客户)?`
- enum展开后: `{SEARCH}意外伤害保险(?:产品|保险)?(?:的客户|客户)?`
- 问题: pattern期望"意外伤害保险"后面是"产品|保险|的客户|客户"，但实际是"名单"

## 解决方案

### 方案1: 调整value_mappings策略（推荐）
**原则**: value_mappings只做必要的归一化，不要过度替换
- ❌ 不要: "意外险"→"意外伤害保险"（改变了词形）
- ✅ 应该: "意外险"→"意外险"（保持原样），或在held_product_category枚举中添加"意外险"
- ❌ 不要: "男性"→"男"（丢失信息）
- ✅ 应该: 规则pattern同时支持"男性"和"男"

### 方案2: 扩展规则pattern（推荐）
为每个规则添加更多常见模式：
```yaml
patterns_template:
  - '{SEARCH}{enum}(?:产品|保险)?(?:的客户|客户)?'
  - '{SEARCH}{enum}(?:产品|保险)?名单'
  - '{SEARCH}{enum}(?:产品|保险)?有哪些'
  - '{SEARCH}{enum}(?:产品|保险)?都有谁'
```

### 方案3: 调整枚举定义
在枚举中同时包含标准值和常见简称：
```yaml
held_product_category:
  - "意外伤害保险"
  - "意外险"  # 添加简称
  - "医疗保险"
  - "医疗险"  # 添加简称
  - "重疾险"  # 添加
```

### 方案4: 移除部分value_mappings
对于会破坏匹配的映射，直接移除：
```yaml
# 移除这些映射
# held_product_category:
#   意外险: "意外伤害保险"  # 移除
#   医疗险: "医疗保险"      # 移除
```

## 建议的优化顺序

1. **立即修复**: 移除破坏性的value_mappings（如"意外险"→"意外伤害保险"）
2. **短期优化**: 为规则添加"名单"、"有哪些"等常见模式
3. **中期优化**: 调整枚举定义，同时包含标准值和简称
4. **长期优化**: 重新设计预处理和匹配策略

## 成功案例分析

### 唯一完全匹配: "未配置重疾险的客户"
- 预处理后: "未配置重疾险的客户"（"重疾险"未被替换）
- 匹配规则: "寿险责任类型-未持有"
- 成功原因: "重疾"在life_liability_type枚举中，且pattern匹配成功

### 部分匹配: "有娃但没配置重疾险的客户"
- 预处理后: "有子女但没配置重疾险的客户"
- 匹配字段: family_members.relationship, life_liability_type
- 成功原因: "娃"→"子女"映射正确，"重疾"匹配成功

### 部分匹配: "父母70岁以上的客户"
- 匹配字段: family_members.age
- 成功原因: 新增的父母年龄规则工作
- 问题: 关系推断规则未触发

## 结论

当前L2规则的主要问题不是规则数量不足，而是：
1. **value_mappings预处理策略不当**，将用户输入转换为规则无法匹配的形式
2. **规则pattern覆盖不足**，缺少"名单"、"有哪些"等常见结尾
3. **枚举定义不完整**，缺少常见简称

优先级最高的修复是：**调整或移除破坏性的value_mappings**。

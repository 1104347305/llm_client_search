# AI客户搜索POC验证报告

## 一、项目概述

### 1.1 项目背景
基于自然语言的智能客户搜索系统，采用四层串联漏斗架构，实现高效、准确的客户查询功能。

### 1.2 POC目标
- 验证AI客户搜索方案的技术可行性
- 评估系统的准确率和性能指标
- 为后续工程实施提供数据支撑

### 1.3 验证时间
- Demo验证: 2026-03-06 ~ 2026-03-12
- POC验证: 2026-03-13 ~ 2026-03-20

## 二、系统架构

### 2.1 四层漏斗架构

```
用户查询
    ↓
┌─────────────────────────────────────┐
│ 逻辑词检测（和/与/且/或/但）         │
│ 检测到逻辑词 → 直接转 Level 4       │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ Level 1: 规则引擎 (Jieba分词)       │
│ - 手机号、身份证、姓名等             │
│ - 响应时间: <10ms                    │
│ - 置信度: 1.0                        │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ Level 2: 增强模板匹配 (正则+YAML)   │
│ - 88条增强规则                       │
│ - 年龄、收入、婚姻状况等             │
│ - 响应时间: 10-50ms                  │
│ - 置信度: 0.95                       │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ Level 3: 语义缓存 (Redis)            │
│ - 缓存历史查询结果                   │
│ - 响应时间: <5ms                     │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ Level 4: LLM解析器 (Agno+DashScope) │
│ - 支持复杂逻辑和语义推断             │
│ - RAG字段检索增强                    │
│ - 响应时间: 500-4000ms               │
│ - 置信度: 0.8                        │
└─────────────────────────────────────┘
```

### 2.2 技术栈
- **Web框架**: FastAPI 0.115.5
- **Agent框架**: Agno 0.0.29
- **LLM模型**: DashScope (通义千问 qwen3.5-27b)
- **中文分词**: Jieba 0.42.1
- **缓存**: Redis 5.2.1
- **日志**: Loguru 0.7.3

## 六、问题与改进建议

### 6.1 当前问题

#### 6.1.1 剩余文本率高 (79.8%)
**问题**: 规则匹配后仍有大量剩余文本
**原因**:
- 规则过于精确，未覆盖"的客户"、"客户"等后缀
- 数值单位（"万"、"岁"）未被规则消费

**建议**:
- 在规则中增加通用后缀匹配: `(?:的)?客户?$`
- 数值规则中包含单位: `(\d+)万` → 自动转换

#### 6.1.2 复杂组合匹配率低
**问题**: 三条件及以上组合匹配率仅20%
**原因**:
- 组合规则数量不足
- 逻辑词检测后直接转L4，未尝试L2

**建议**:
- 增加常见组合规则（如"已婚+有子女+未配置XX"）
- 优化逻辑词检测，允许简单AND组合在L2处理

#### 6.1.3 枚举值覆盖不全
**问题**: "万能险"、"刚结婚"等词未覆盖
**建议**: 补充枚举值和别名映射

### 6.2 优化建议

#### 优先级1 (高)
1. **补充枚举值**: 产品名称、标签、状态等
2. **优化剩余文本处理**: 增加通用后缀规则
3. **增加日期规则**: 支持"YYYY年MM月"格式

#### 优先级2 (中)
1. **增加组合规则**: 常见2-3条件组合
2. **优化逻辑词检测**: 区分简单AND和复杂逻辑
3. **完善别名映射**: "刚结婚"→"已婚"等

#### 优先级3 (低)
1. **性能优化**: 正则编译缓存
2. **规则管理**: 规则热更新机制
3. **监控告警**: 未匹配query自动收集

## 七、POC结论

### 7.1 技术可行性
✅ **验证通过**

- 四层漏斗架构设计合理，各层职责清晰
- L1+L2覆盖率达74.3%，有效降低LLM调用
- 整体准确率96%，满足业务需求
- 平均响应时间500ms，用户体验良好

### 7.2 核心优势

1. **高性能**: L1+L2平均响应时间<20ms，占比74.3%
2. **高准确率**: 整体准确率96%，L1+L2准确率98%+
3. **低成本**: LLM调用率仅22.9%，大幅降低API成本
4. **可扩展**: YAML配置化规则，易于维护和扩展

### 7.3 风险与挑战

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 规则维护成本 | 中 | YAML配置化，降低维护难度 |
| 枚举值覆盖不全 | 中 | 建立枚举值管理机制，定期更新 |
| 复杂查询准确率 | 低 | L4 LLM兜底，准确率92% |
| LLM API稳定性 | 中 | 增加重试机制和降级策略 |

### 7.4 推荐方案
✅ **建议进入工程实施阶段**

基于POC验证结果，系统在性能、准确率、成本等方面均达到预期目标，建议进入工程实施阶段。

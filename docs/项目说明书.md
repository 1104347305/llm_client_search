## 1. 项目概述

**客户智能搜索系统**是面向保险代理人的自然语言客户检索平台。代理人可直接用中文描述查询意图，系统自动将其转换为结构化查询条件，调用后端搜索 API 返回匹配客户。

### 核心能力

| 能力 | 说明 |
|------|------|
| 自然语言理解 | 将中文查询解析为字段+操作符+值的结构化条件 |
| 多维条件组合 | 支持 AND / OR 逻辑，支持嵌套字段（保单、家庭成员、证件等） |
| 规则优先、LLM 兜底 | 确定性规则处理 95% 场景，LLM 处理长尾 |
| RAG 字段召回 | Elasticsearch 全文检索字段意图，辅助 LLM 生成准确条件 |
| AgentOS 控制台 | 接入 os.agno.com，提供对话式搜索界面 |

### 典型查询示例

```
45岁以上已婚年收入20万以上的客户
本月生日的客户
未配置重疾险的高净值客户
下个月需要缴费的客户
保单号P644037的客户
居家等级V1以上的客户
最近1年内承保的客户
1953年出生的客户
```

## 2. 系统架构

```
用户输入（自然语言）
        │
        ▼

│           Query Router              │
│   （四层串行漏斗，短路优化）         │
│                                     │
│  L1 规则引擎  ──►  L2 增强匹配器   │
│       │                  │          │
│       └──────┬───────────┘          │
│              ▼                      │
│         L3 语义缓存（Redis）        │
│              │                      │
│              ▼（未命中时）          │
│         L4 LLM 解析器               │
└─────────────────────────────────────┘
        │
        ▼
  结构化查询条件（Conditions）
        │
        ▼
┌─────────────────────┐    ┌──────────────────────┐
│  Search API Client  │───►│  后端搜索服务 :8001  │
└─────────────────────┘    └──────────────────────┘
        │
        ▼

```

### 服务端口

| 服务 | 端口 | 说明 |
|------|------|------|
| 搜索 API (FastAPI) | 8000 | REST 接口，含自然语言搜索和结构化搜索 |
| AgentOS | 7777 | Agno 对话式 Agent 服务，供 os.agno.com 连接 |
| Elasticsearch | 9200 | 字段意图 RAG 检索 |

| Redis | 6379 | 语义缓存（可选） |

## 3. 四层漏斗路由详解

系统采用**串行漏斗**设计：优先使用低成本确定性方法，仅在前层无法解析时才调用下一层。

### L1 — 规则引擎（确定性实体提取）

- **方法**：Jieba 分词 + 正则规则
- **覆盖场景**：人名、手机号等高置信度实体
- **置信度**：1.0
- **触发条件**：始终执行

### L2 — 增强模板匹配器（YAML 驱动规则）

- **方法**：YAML 配置的正则表达式 + 枚举展开 + 别名归一化
- **覆盖场景**：年龄、婚姻、险种、VIP 等级、缴费日期等 150+ 规则
- **置信度**：0.95
- **触发条件**：始终执行，与 L1 结果合并

### L3 — 语义缓存（Redis 向量检索）

- **方法**：向量相似度匹配历史缓存
- **覆盖场景**：重复或相似查询
- **置信度**：继承缓存命中时的原始置信度
- **触发条件**：L1+L2 条件为空时检查

### L4 — LLM 解析器（AI 兜底）

- **方法**：调用 Qwen 3.5-27b，结合 RAG 字段召回的动态 prompt
- **覆盖场景**：长尾、复杂、模糊查询
- **置信度**：0.7（LLM 输出不保证）
- **触发条件**：L1+L2+L3 均无结果时调用

### 路由决策流程

```
query
  │
  ├─ 检测复杂逻辑运算符（AND/OR 混合）→ 直接跳 L4
  │
  ├─ L1 提取（始终）
  ├─ L2 匹配（始终）
  │
  ├─ L1+L2 有结果？→ 返回（matched_level=2）
  │
  ├─ L3 缓存命中？→ 返回（matched_level=3）
  │
  └─ 调用 L4 LLM → 返回（matched_level=4）
```

## 4. 核心组件说明

### 4.1 QueryRouter (`app/core/query_router.py`)

统一入口，协调四层调用。核心方法：

```python
result: ParsedQuery = await router.route_with_peeling(query)
# result.conditions   → 结构化条件列表
# result.matched_level → 命中层级 (1-4)
# result.confidence   → 置信度
```

### 4.2 Level2EnhancedMatcher (`app/core/level2_enhanced_matcher.py`)

基于 YAML 配置的正则匹配引擎。详见《规则引擎 L2 使用说明书》。

### 4.3 FieldRegistry (`app/core/field_registry.py`)

字段意图 RAG 检索，基于 Elasticsearch。

- 加载 `config/field_definitions.yaml`（145 个意图）
- 写入 ES 索引 `field_intents`，使用 BM25 检索
- 检索结果注入 L4 LLM prompt，提升字段识别准确率

### 4.4 SearchAPIClient (`app/services/search_api_client.py`)

封装下游搜索服务调用，将结构化 `SearchRequest` POST 到 `http://localhost:8001`。

### 4.5 数据模型 (`app/models/schemas.py`)

```python
class Condition:
    field: str          # 字段名，如 "age", "policies.policy_id"
    operator: Operator  # MATCH / GTE / LTE / RANGE / CONTAINS / NOT_CONTAINS / EXISTS / NOT_EXISTS / NESTED_MATCH
    value: ...          # str | int | float | RangeValue | List[str] | None

class RangeValue:
    min: str | int | float | None   # 支持数值和日期字符串
    max: str | int | float | None
```

## 5. API 接口说明

Base URL：`http://localhost:8000/api/v1`

### 5.1 自然语言搜索

```
POST /search/natural
```

**请求体**

```json
{
  "query": "45岁以上已婚的客户",
  "agent_id": "A000000",
  "page": 1,
  "size": 20
}
```

**响应**

```json
{
  "success": true,
  "data": { "total": 128, "list": [...] },
  "matched_level": 2,
  "confidence": 0.95,
  "conditions": [
    { "field": "age", "operator": "GTE", "value": 45 },
    { "field": "marital_status", "operator": "MATCH", "value": "已婚" }
  ]
}
```

### 5.2 结构化搜索

```
POST /search/structured
```

```json
{
  "header": { "agent_id": "A000000", "page": 1, "size": 20 },
  "query_logic": "AND",
  "conditions": [
    { "field": "age", "operator": "GTE", "value": 45 }
  ]
}
```

### 5.3 字段 RAG 检索

```
POST /fields/retrieve
```

```json
{ "query": "重疾险未配置", "top_k": 6 }
```

**响应**：返回相关字段意图 + 格式化的 LLM prompt 片段。

### 5.4 字段索引重建

```
POST /fields/reindex
```

修改 `config/field_definitions.yaml` 后调用此接口，热更新 ES 索引，无需重启服务。

### 5.5 健康检查

```
GET /health
```

## 6. 配置文件说明

### 6.1 `config/settings.py`

通过环境变量或 `.env` 文件覆盖默认值：

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `LLM_MODEL` | `qwen3.5-27b` | LLM 模型 |
| `LLM_API_KEY` | — | Aliyun DashScope API Key |
| `LLM_BASE_URL` | DashScope 地址 | 兼容 OpenAI 格式 |
| `SEARCH_API_BASE_URL` | `http://localhost:8001` | 下游搜索服务 |
| `ES_HOST` | `http://localhost:9200` | Elasticsearch 地址 |
| `ES_ANALYZER` | `ik_max_word` | 中文分词器（需安装 IK 插件） |
| `AGNO_API_KEY` | — | os.agno.com 控制台 API Key |

### 6.2 `config/enhanced_rules.yaml`

L2 规则引擎配置，详见《规则引擎 L2 使用说明书》。

### 6.3 `config/field_definitions.yaml`

RAG 知识库，145 个字段意图定义，供 L4 LLM 检索使用。每个意图包含：
- `retrieval_text`：检索关键词
- `field`：ES 字段名
- `operator`：操作符
- `enum`：枚举值（可选）
- `examples`：示例查询（辅助 LLM 理解）

## 7. AgentOS 控制台

### 连接方式

1. 启动 `agent_os_app.py`（端口 7777）
2. 打开 [https://os.agno.com](https://os.agno.com)
3. 添加端点：`http://localhost:7777`
4. 刷新页面后即可对话

### 工具说明

| 工具 | 功能 |
|------|------|
| `search_customers` | 自然语言搜索客户，默认展示9个关键字段+查询条件涉及字段 |
| `retrieve_fields` | 查看某个查询会用到哪些字段定义 |

### 展示字段控制

- 默认展示：`name, mobile_phone, age, gender, marital_status, life_insurance_vip, customer_value, customer_temperature, operation_stage` + 查询条件相关字段
- 自定义：在 `display_fields` 参数中指定逗号分隔的字段名

## 8. 部署与启动

### 快速启动

```bash

./start.sh

# 仅启动搜索 API
./start.sh --api-only

# 仅启动 AgentOS
./start.sh --agent-only

./start.sh --stop
```

```bash
cp .env.example .env
```

## 10. 常见问题

**Q：结果不稳定，每次查询结果不同？**

A：查询进入了 L4 LLM，LLM 有随机性。应为该查询在 L2 添加对应规则，将其固化为确定性匹配。

**Q：搜索结果为空？**

A：可能原因：① 下游搜索服务 (:8001) 未启动；② 查询条件过严，建议放宽条件后重试。

**Q：修改 YAML 规则后不生效？**

A：搜索 API 以 `--reload` 模式启动，修改文件后自动热重载。AgentOS 需重启：`./start.sh --stop && ./start.sh --agent-only`。

**Q：字段意图 RAG 检索效果差？**

A：建议安装 Elasticsearch IK 中文分词插件后，调用 `POST /api/v1/fields/reindex` 重建索引。

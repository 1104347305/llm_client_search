# 地址搜索方案

## 一、目标场景

| 编号 | 场景 | 示例查询 | 期望条件 |
|------|------|----------|----------|
| A | 纯地址匹配 | "天通苑小区的客户" | `customerAddress MATCH "天通苑"` |
| B | 地名+半径内 | "天通苑周围1公里以内" | `customerLocation GEO_RADIUS {place_name:"天通苑", radius:1000}` |
| C | 地名+半径外 | "天通苑周围1公里以外" | `customerLocation NOT_GEO_RADIUS {place_name:"天通苑", radius:1000}` |
| D | 我周围+半径内 | "我周围1公里以内" | `customerLocation GEO_RADIUS {place_name:null, radius:1000}` |
| E | 我周围+半径外 | "我周围1公里以外" | `customerLocation NOT_GEO_RADIUS {place_name:null, radius:1000}` |

**`place_name=null` 的语义**：以客户自身存储的地址坐标为中心，由后端 8001 自行取值。

---

## 二、架构总览

**核心原则：本服务只做意图识别+参数提取，不做地理编码。后端 8001 负责地名→坐标转换。**

```
用户查询 → L2规则引擎(本服务8000) → 提取 place_name + radius → Condition(GEO_RADIUS)
                │
                │ 不做地理编码，不调外部API
                │
                ↓
       search_api_client 序列化 → POST /api/v1/search/customer → 后端8001
                │
                │ 后端8001职责：
                │   - place_name 有值 → 自行地理编码(高德/百度) → 经纬度
                │   - place_name=null  → 取客户定位的坐标
                │   - 构造 ES geo_distance 查询
                ↓
        ES geo_distance 查询 → 返回结果
```

## 三、后端搜索服务（8001）需要准备的内容

### 3.1 数据存储要求

| 字段 | 类型 | 说明 |
|------|------|------|
| `customerAddress` | text | 客户地址文本，支持 MATCH / CONTAINS |
| `customerLocation` | geo_point（推荐，ES 原生） | 客户地址经纬度，`{lat, lon}` |

建议：数据入库时对 `customerAddress` 做一次地理编码，将结果写入 `customerLocation`。不做实时地理编码。

### 3.2 后端 API 协议扩展

#### 3.2.1 协议版本

新增操作符 `GEO_RADIUS` 和 `NOT_GEO_RADIUS`，协议版本从 V3 升级为 **V4**，其他协议字段不变。

#### 3.2.2 condition 格式

**精确地址匹配（MATCH）— 不变：**

```json
{
  "field": "customerAddress",
  "operator": "MATCH",
  "value": "天通苑"
}
```

**半径内含（GEO_RADIUS）— place_name 有值：**

```json
{
  "field": "customerLocation",
  "operator": "GEO_RADIUS",
  "value": {
    "place_name": "天通苑",
    "radius": 1000
  }
}
```

**半径内含（GEO_RADIUS）— place_name=null（"我周围"）：**

```json
{
  "field": "customerLocation",
  "operator": "GEO_RADIUS",
  "value": {
    "place_name": null,
    "radius": 1000
  }
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `place_name` | string 或 null | 地名（需后端自行地理编码）；null 表示以客户自身存储的地址坐标为中心 |
| `radius` | int | 半径，单位：米 |

**半径外不含（NOT_GEO_RADIUS）：**

```json
{
  "field": "customerLocation",
  "operator": "NOT_GEO_RADIUS",
  "value": {
    "place_name": "天通苑",
    "radius": 1000
  }
}
```

### 3.3 condition 处理逻辑（伪代码）

```python
def handle_geo_radius(condition, customer_data):
    place_name = condition["value"]["place_name"]
    radius = condition["value"]["radius"]       # 单位：米，可能为 None

    if place_name is not None:
        # 模式一：地名 → 自行地理编码 → 经纬度
        lat, lng = geocode(place_name)           # 调高德/百度 API
    else:
        # 模式二：取客户存储的坐标
        lat = customer_data["customerLocation"]["lat"]
        lng = customer_data["customerLocation"]["lon"]
        # 若客户坐标为空 → 跳过此条件，或在结果中降级

    if radius is not None:
        # ES geo_distance 查询
        es_query = {
            "geo_distance": {
                "distance": f"{radius}m",
                "customerLocation": {"lat": lat, "lon": lng}
            }
        }
    else:
        # 无半径 → 降级为文本 MATCH（或使用默认半径）
        es_query = {"match": {"customerAddress": place_name}}

    if operator == "NOT_GEO_RADIUS":
        es_query = {"bool": {"must_not": es_query}}

    return es_query
```

### 3.5 后端需要做的事清单

| # | 事项 | 说明 |
|---|------|------|
| 1 | 确认或新增 `customerAddress` 文本字段 | MATCH / CONTAINS 用 |
| 2 | 确认或新增 `customerLocation` geo_point 字段 | GEO_RADIUS / NOT_GEO_RADIUS 用 |
| 3 | 历史数据回填经纬度 | 对已有地址文本做地理编码（一次性） |
| 4 | 新数据写入流程 | 新客户创建时对地址做地理编码 |
| 5 | 支持 `GEO_RADIUS` 操作符 | 解析 value → 地理编码（如有 place_name） → ES geo_distance |
| 6 | 支持 `NOT_GEO_RADIUS` 操作符 | 同上，放入 ES `bool.must_not` |
| 7 | 返回 `distance` 字段（可选） | 实际距中心点距离，前端展示用 |

## 四、本服务改动清单

### 4.1 数据模型（models/schemas.py）

```python
# === 新增操作符 ===
class Operator(str, Enum):
    # ... 现有 10 个保持不变 ...
    GEO_RADIUS = "GEO_RADIUS"          # 地理半径内含
    NOT_GEO_RADIUS = "NOT_GEO_RADIUS"  # 地理半径外不含

# === 新增值类型 ===
class GeoRadiusValue(BaseModel):
    """地理半径值 — 本服务只提取地名和半径，不做地理编码"""
    place_name: Optional[str] = None  # null = 以客户自身地址坐标为中心
    radius: Optional[int] = None      # 单位：米，null = 无半径限制

# === Condition.value Union 扩展 ===
class Condition(BaseModel):
    value: Optional[Union[
        str, int, float, RangeValue, GeoRadiusValue,  # ← 新增
        Dict[str, Any], List[str]
    ]] = None
```

### 4.6 L2 引擎扩展（steps/level2_enhanced_matcher.py）

在 `_extract_value` 方法中新增两种 `value_type`：

```python
elif value_type == "geo_radius_place":
    # 从捕获组提取地名和半径，不做地理编码
    place_group = value_config.get("place_group", 1)
    radius_group = value_config.get("radius_group", 2)
    unit = value_config.get("unit", "km")
    try:
        place = match.group(place_group)
        radius_str = match.group(radius_group)
        if not place or not radius_str:
            return None
        radius = int(radius_str)
        if unit == "km":
            radius *= 1000
        return GeoRadiusValue(place_name=place, radius=radius)
    except (IndexError, ValueError):
        return None

elif value_type == "geo_radius_self":
    # 只有半径，place_name=null（表示客户自身地址坐标）
    radius_group = value_config.get("radius_group", 1)
    unit = value_config.get("unit", "km")
    try:
        radius_str = match.group(radius_group)
        if not radius_str:
            return None
        radius = int(radius_str)
        if unit == "km":
            radius *= 1000
        return GeoRadiusValue(place_name=None, radius=radius)
    except (IndexError, ValueError):
        return None
```

也需在 `_get_operator` 中确保 `GEO_RADIUS` / `NOT_GEO_RADIUS` 操作符被正确识别。

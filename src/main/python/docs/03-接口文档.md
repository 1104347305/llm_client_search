# 03-接口文档

服务默认：

```text
host: 0.0.0.0
port: 8000
prefix: /api/v1
```

## 健康检查

```http
GET /health
```

说明：

- 检查 `QueryRouter` 是否加载完成。
- 检查解析端点是否 ready。
- 未 ready 时返回 HTTP 503。

## 解析查询条件

```http
POST /api/v1/client_search_query_parse
```

用途：解析自然语言查询，不执行搜索。

请求示例：

```json
{
  "source": "askbob",
  "user_text": "45岁以上且买过重疾险的客户",
  "session_id": "s-001",
  "trace_id": "t-001",
  "user_id": "u-001",
  "ts": 1710000000,
  "user_action": "write",
  "action_scenario": "customerSearch",
  "extra_input_params": {}
}
```

响应示例：

```json
{
  "code": 0,
  "msg": "操作成功",
  "data": {
    "robot_text": "查询意图摘要",
    "end_flag": 1,
    "trace_id": "t-001",
    "extra_output_params": {
      "query": "45岁以上且买过重疾险的客户",
      "query_logic": "AND",
      "conditions": [
        {
          "field": "clientBirthday",
          "operator": "LTE",
          "value": "1981-12-31 00:00:00"
        }
      ],
      "matched_level": 2,
      "rewritten_query": "45岁以上且买过重疾险的客户",
      "matched_patterns": "...",
      "cost_times": 123,
      "confidence": 0.95,
      "intent_summary": "查询意图摘要"
    }
  }
}
```

说明：

- `matched_level=1`：L1 命中。
- `matched_level=2`：L2 命中。
- `matched_level=4`：L4 LLM 兜底命中。
- `matched_level=0`：未解析出条件。
- 若 `ENABLE_PARSE_RESPONSE_AES=true`，`robot_text` 和 `extra_output_params` 会加密。

## 解析查询条件，强制明文

```http
POST /api/v1/client_search_query_parse_no_encipher
```

用途同 `/client_search_query_parse`，但固定返回明文。建议开发联调优先使用这个接口。

curl 示例：

```bash
curl -X POST http://localhost:8000/api/v1/client_search_query_parse_no_encipher \
  -H "Content-Type: application/json" \
  -d '{
    "source": "askbob",
    "user_text": "45岁以上且买过重疾险的客户",
    "session_id": "debug-session",
    "trace_id": "debug-trace",
    "user_id": "debug-user",
    "user_action": "write",
    "action_scenario": "customerSearch",
    "extra_input_params": {}
  }'
```

## 全量配置热更新

```http
POST /api/v1/config/reload
```

请求：

```json
{
  "force_reindex_fields": false,
  "wait": false
}
```

字段说明：

| 字段 | 说明 |
| --- | --- |
| `force_reindex_fields` | 是否同步重建字段意图 ES 索引 |
| `wait` | 是否等待当前 worker 刷新完成 |

运行机制：

- 热更新采用后台构建新运行时、成功后原子切换的方式。
- 热更新期间，解析接口继续使用当前已加载的旧 `QueryRouter`，不等待新运行时构建完成。
- 新运行时完整构建成功后，才切换 `FieldRegistry`、`QueryRouter`、intent summary service 等内存单例。
- 如果热更新失败，会保留旧运行时继续服务，并在 runtime reload 状态中记录错误。
- `wait=true` 只等待接收该请求的当前 worker 完成刷新；多 worker 仍通过 `reload_marker` 通知其它 worker 后台刷新。

返回包含：

- `success`
- `message`
- `started`
- `force_reindex_fields`
- runtime reload 状态
- `reload_marker`

## 查看可热刷新的配置文件

```http
GET /api/v1/config/reload/files
```

返回可用于按文件热刷新的 alias、路径与刷新范围。

常用 alias：

```text
runtime_config
field_definitions
enhanced_rules
value_mappings
intent_summary
field_mapping
field_enums
```

枚举文件也会注册自己的 alias，例如各个 `*_enums_args.yaml` 的 stem。

## 按文件热更新配置

```http
POST /api/v1/config/reload/files
```

请求：

```json
{
  "files": ["enhanced_rules", "field_enums"],
  "force_reindex_fields": false,
  "wait": true
}
```

`files` 支持：

- alias，例如 `enhanced_rules`
- 文件名，例如 `enhanced_rules_args.yaml`
- 绝对路径，必须命中白名单
- `all`，等同全量刷新

## 重建字段意图 ES 索引

```http
POST /api/v1/fields/reindex
```

用途：

- 修改 `field_definitions_args.yaml` 后，重建 ES 字段意图索引。
- 更新影响 L4 RAG 的字段知识后建议调用。

## 字段意图检索调试

```http
POST /api/v1/fields/retrieve
```

请求：

```json
{
  "query": "买过重疾险的客户",
  "top_k": 8
}
```

响应：

```json
{
  "query": "买过重疾险的客户",
  "total": 3,
  "intents": [
    {
      "id": "...",
      "field": "...",
      "operator": "CONTAINS",
      "value_type": "enum",
      "retrieval_text": "...",
      "enum": ["..."],
      "unit": null,
      "notes": "...",
      "examples": []
    }
  ],
  "prompt_section": "..."
}
```

用途：

- 检查新字段是否能被 L4 RAG 召回。
- 查看注入 LLM prompt 的字段 schema。
- 排查 L4 不识别某类查询的问题。

## 下游搜索接口状态

代码中存在：

```text
services/search_service.py
services/search_api_client.py
```

它们会调用：

```http
POST {SEARCH_API_BASE_URL}/api/v1/search/customer
```

但当前 FastAPI router 没有暴露自然语言搜索或结构化搜索接口。当前主对外能力是“解析不搜索”。如需恢复搜索接口，应新增 route，并补请求日志、异常处理和测试。

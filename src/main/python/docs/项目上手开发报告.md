# llm_client_search 项目上手开发报告

更新时间：2026-05-13

## 1. 项目定位

`llm_client_search` 是一个“自然语言客户搜索条件解析”服务。它接收用户输入的中文查询，例如“45岁以上且买过重疾险的客户”，将其转换为统一的结构化查询条件，供下游客户搜索 API 执行检索。

当前代码主目录已经迁移到：

```text
src/main/python/
```

项目采用 FastAPI 提供 HTTP 接口，核心能力是四层解析链路：

```text
用户 query
  -> QueryRouter.route_with_peeling
  -> L1 正则/确定性实体
  -> L2 YAML 增强规则
  -> L3 语义缓存（代码存在但当前主链路注释未启用）
  -> L4 LLM + RAG 字段意图兜底
  -> Condition / QueryLogic / Intent Summary
```

## 3. 核心解析流程

### 3.1 主链路

核心类：`steps/query_router.py` 的 `QueryRouter`

主方法：

```python
await QueryRouter().route_with_peeling(query)
```

处理顺序：

1. 去除空格。
2. 通过 `FieldRegistry.normalize_query()` 对口语值做归一化。
3. 记录裸姓名/裸值候选。
4. 执行 L1 确定性实体提取。
5. 执行 L2 YAML 规则匹配和候选召回。
6. 如果 L2 命中，则丢弃 L1 条件，避免 L1 粗粒度结果干扰 L2。
7. L3 当前主链路代码被注释，没有实际查询缓存。
8. 如果 L1/L2/L3 都没有条件，则：
   - 裸值弱命中时返回 OR 候选字段；
   - 否则进入 L4 LLM 解析。
9. 对条件做字段合法性校验、重复去重、EXISTS 冲突清理、单值 CONTAINS 转 MATCH、范围合并等后处理。

### 3.2 L1：确定性规则

文件：`steps/level1_rule_engine.py`

支持：

- 客户号：`C` + 15 位数字
- 保单号：`P` + 15 位数字
- 身份证号：18 位
- 手机号：11 位大陆手机号
- Jieba 人名识别

输出字段通过 `models/field_mapping.py` 从 `field_mapping_args.yaml` 的 `query_fields` 获取，避免硬编码真实字段名。

### 3.3 L2：增强规则

文件：`steps/level2_enhanced_matcher.py`

配置文件：`config/enhanced_rules_args.yaml`

当前规模：

- 基础规则：417 条
- 组合规则：70 条
- pattern vars：2 组

能力：

- 正则规则匹配。
- `{enum}` / `{multi_enum}` 根据枚举配置自动展开。
- 口语值归一化，例如“男士/男性/男”映射到标准枚举。
- 组合规则引用。
- 裸值弱匹配，将无法确认字段的裸值扩成候选字段 OR。
- L4 RAG 可复用 L2 的候选召回结果。

### 3.4 L3：语义缓存

文件：`steps/level3_semantic_cache.py`

运行配置里仍有 Redis 与 L3 开关：

```yaml
ENABLE_L3: true
REDIS_HOST: localhost
REDIS_PORT: 6379
CACHE_TTL: 3600
SEMANTIC_CACHE_THRESHOLD: 0.85
```

但 `QueryRouter.route_with_peeling()` 中 Level 3 相关代码当前被注释，因此主流程没有实际使用 Redis 缓存。后续如果恢复 L3，需要同步补测试和热更新行为。

### 3.5 L4：LLM + RAG 兜底

文件：`steps/level4_llm_parser.py`

当前实现使用 OpenAI 兼容异步客户端：

```python
AsyncOpenAI(api_key=settings.LLM_API_KEY, base_url=settings.LLM_BASE_URL)
```

RAG 三路召回：

1. ES BM25：`FieldRegistry.retrieve(query, top_k)`
2. Trie 枚举精确匹配：`FieldRegistry.retrieve_by_enum(query)`
3. L2 候选召回：`Level2EnhancedMatcher.recall_candidates()`

三路结果按 `field + operator + intent_id` 聚合后注入 prompt。若 RAG 无召回，L4 会跳过并返回空条件，避免 LLM 幻觉字段。

## 4. Schema 与字段定义如何查

用户提到的 “scheam” 建议按以下几类查。

### 4.1 接口 schema

文件：`models/schemas.py`

核心模型：

- `ParseApiRequest`：AskBob 标准协议入参。
- `ParseApiResponse`：AskBob 标准协议出参。
- `ParseApiExtraOutput`：解析结果详情。
- `Condition`：单个搜索条件。
- `RangeValue`：范围值。
- `QueryLogic`：`AND` / `OR`。
- `Operator`：`MATCH`、`GT`、`GTE`、`LT`、`LTE`、`RANGE`、`CONTAINS`、`NOT_CONTAINS`、`EXISTS`、`NOT_EXISTS`。
- `SearchRequest` / `NaturalLanguageSearchRequest` / `SearchResponse`：内部搜索服务封装使用。

`Condition.value` 的约束：

- `CONTAINS` / `NOT_CONTAINS`：自动规范为列表。
- 其他 operator 如果收到列表，会取第一个值。
- `RANGE` 推荐使用 `{ "min": ..., "max": ... }`。
- `EXISTS` / `NOT_EXISTS` 不需要传 `value`。

### 4.2 字段意图 schema

文件：`config/field_definitions_args.yaml`

当前字段意图数量：179 条。

典型字段意图结构：

```yaml
intents:
  - id: xxx
    field: clientAge
    operator: GTE
    value_type: number
    retrieval_text: ...
    enum: [...]
    unit: 岁
    notes: ...
    examples: [...]
    negative_examples: [...]
```

用途：

- 提供合法字段清单。
- 支撑 L4 RAG 检索。
- 作为 `QueryRouter._validate_conditions()` 的字段校验基准。
- 通过 `/api/v1/fields/retrieve` 调试召回结果。

### 4.3 字段映射

文件：`config/field_mapping_args.yaml`

当前规模：

- `query_fields`：12 个常用业务字段别名。
- `field_context_groups`：2 组上下文分组。

代码入口：`models/field_mapping.py`

常用方法：

- `get_query_field("customer_name")`
- `get_field_context_group("...")`
- `get_sensitive_field_group("...")`
- `get_name_candidate_values("...")`

### 4.4 枚举和值归一化

主要文件：

- `config/field_enums_args.yaml`：34 个普通枚举字段。
- `config/*_enums_args.yaml`：产品名、职业名等大枚举。
- `config/value_mappings_args.yaml`：23 个字段的口语映射。

更新枚举后要关注：

1. L2 的 `{enum}` 正则展开是否覆盖新值。
2. L4 的 Trie 枚举召回是否能命中新值。
3. 是否需要调用 `/api/v1/fields/reindex` 更新 ES 字段意图索引。

### 5.1 健康检查

### 5.2 解析查询条件，加密可选

```http
POST /api/v1/client_search_query_parse
```

用途：解析自然语言查询，不执行搜索。

请求：

```json
{
  "source": "askbob",
  "user_text": "45岁以上且买过重疾险的客户",
  "session_id": "s-001",
  "trace_id": "t-001",
  "user_id": "u-001",
  "ts": 1710000000,
  "user_action": "write",
  "action_scenario": "customerSearch",
  "extra_input_params": {}
}
```

响应：

```json
{
  "code": 0,
  "msg": "操作成功",
  "data": {
    "robot_text": "查询意图摘要",
    "end_flag": 1,
    "trace_id": "t-001",
    "extra_output_params": {
      "query": "45岁以上且买过重疾险的客户",
      "query_logic": "AND",
      "conditions": [
        {
          "field": "clientBirthday",
          "operator": "LTE",
          "value": "1981-12-31 00:00:00"
        }
      ],
      "matched_level": 2,
      "rewritten_query": "45岁以上且买过重疾险的客户",
      "matched_patterns": "...",
      "cost_times": 123,
      "confidence": 0.95,
      "intent_summary": "查询意图摘要"
    }
  }
}
```

加密行为：

- 由 `ENABLE_PARSE_RESPONSE_AES` 控制。
- 开启后 `robot_text` 和 `extra_output_params` 会被 `response_crypto.py` 加密。
- AES key 来自 `PARSE_RESPONSE_AES_KEY`，必须满足代码校验长度。

### 5.3 解析查询条件，强制不加密

```http
POST /api/v1/client_search_query_parse_no_encipher
```

用途同上，但忽略 `ENABLE_PARSE_RESPONSE_AES`，固定返回明文。适合联调和排查。

### 5.4 热更新全量配置

```http
POST /api/v1/config/reload
```

请求：

```json
{
  "force_reindex_fields": false,
  "wait": false
}
```

说明：

- 默认后台刷新，不阻塞请求。
- `wait=true` 时等待当前 worker 完成刷新。
- `force_reindex_fields=true` 时同步重建字段意图索引。
- 会写入 `.client_search_runtime_reload.json` marker，用于多 worker 感知配置更新。

### 5.5 查看可按文件热刷新的配置

```http
GET /api/v1/config/reload/files
```

返回可选 alias、绝对路径和刷新范围。

常用 alias：

- `runtime_config`
- `field_definitions`
- `enhanced_rules`
- `value_mappings`
- `intent_summary`
- `field_mapping`
- `field_enums`
- 各个 `*_enums_args.yaml` 的 stem 名称

### 5.6 按文件热更新配置

```http
POST /api/v1/config/reload/files
```

请求：

```json
{
  "files": ["enhanced_rules", "field_enums"],
  "force_reindex_fields": false,
  "wait": true
}
```

说明：

- `files` 支持 alias、文件名、绝对路径。
- `files: ["all"]` 等同全量刷新。
- 只有命中白名单的配置文件允许刷新。

### 5.7 重建字段意图 ES 索引

```http
POST /api/v1/fields/reindex
```

用途：

- 更新 `field_definitions_args.yaml` 后重建 ES 字段意图索引。
- 更新影响 L4 RAG 的字段知识时建议调用。

### 5.8 字段意图检索调试

```http
POST /api/v1/fields/retrieve
```

请求：

```json
{
  "query": "买过重疾险的客户",
  "top_k": 8
}
```

响应字段：

- `query`
- `total`
- `intents`
- `prompt_section`

用途：

- 调试 L4 能看到哪些字段 schema。
- 排查新字段为什么没有进入 prompt。
- 检查 ES/Trie/RAG 召回质量。

### 5.9 搜索服务封装状态

代码中存在 `SearchService.natural_language_search()` 和 `SearchService.structured_search()`，会调用下游：

```http
POST {SEARCH_API_BASE_URL}/api/v1/search/customer
```

但当前 `api/client_search_query_parse_post.py` 注册的路由没有暴露自然语言搜索或结构化搜索接口，主对外接口是“解析不搜索”。如果需要恢复搜索接口，需要新增 FastAPI route 并补上请求日志、错误处理和测试。

## 8. 新人开发建议路径

### 8.1 新增一个普通字段解析

1. 在 `config/field_definitions_args.yaml` 添加字段 intent。
2. 如字段有枚举，在 `field_enums_args.yaml` 或独立 `*_enums_args.yaml` 添加枚举。
3. 如用户会说口语值，在 `value_mappings_args.yaml` 添加归一化。
4. 如希望 L2 直接命中，在 `enhanced_rules_args.yaml` 添加规则。
5. 调用：

   ```bash
   curl -X POST http://localhost:8000/api/v1/config/reload/files \
     -H "Content-Type: application/json" \
     -d '{"files":["field_definitions","enhanced_rules","field_enums","value_mappings"],"wait":true}'
   ```

6. 如依赖 ES RAG，调用：

   ```bash
   curl -X POST http://localhost:8000/api/v1/fields/reindex
   ```

7. 用 `/api/v1/fields/retrieve` 验证字段能否召回。
8. 用 `/api/v1/client_search_query_parse_no_encipher` 验证最终 conditions。
9. 添加或修复对应测试。

### 8.2 新增一个 L2 规则

重点看 `enhanced_rules_args.yaml` 中已有规则格式。

一般规则包含：

```yaml
- name: "规则名称"
  field: "字段名"
  operator: "MATCH"
  value_type: "capture"
  patterns:
    - "正则表达式"
```

枚举规则通常使用：

```yaml
enum_ref: "字段名"
patterns_template:
  - "{enum}"
  - "买过{enum}"
```

开发要点：

- 正则尽量收窄上下文，避免误召回。
- 多枚举值查询使用 `{multi_enum}`。
- 对否定语义使用 `NOT_CONTAINS` 或 `NOT_EXISTS`。
- 新增后跑相关 L2 单测。

### 8.3 调试 L4

建议顺序：

1. 调 `/api/v1/fields/retrieve` 看 RAG 是否召回正确字段。
2. 看返回的 `prompt_section` 是否包含 operator、enum、examples。
3. 调 `/api/v1/client_search_query_parse_no_encipher` 看 `matched_level` 是否为 4。
4. 查看 `matched_patterns` 中的 `L4_PROMPT`。
5. 若 RAG 没召回，优先补 `field_definitions_args.yaml` 的 `retrieval_text/examples/notes`，而不是直接改 prompt。

### 8.4 修复测试的优先级

建议按这个顺序推进：

1. 统一测试 import 路径，改为 `src.main.python.steps.*`、`src.main.python.models.*` 等新路径，或增加兼容 shim。
2. 修复 `test_level1.py` 和 `test_logic_operators.py` 的语法错误。
3. 删除或重写引用已不存在模块的测试，例如 `level2_template_matcher`。
4. 将需要 ES/LLM 的测试 mock 掉外部调用，避免 CI 依赖外部服务。
5. 再跑全量 `python -m pytest src/main/python/tests -q`。

## 10. 交接结论

当前项目的主干能力已经集中在“自然语言解析为搜索条件”，并且字段、枚举、规则、摘要、热更新都已经配置化。新同学最快的上手方式是：

1. 先看 `models/schemas.py` 理解出参结构。
2. 再看 `steps/query_router.py` 理解 L1/L2/L4 如何串起来。
3. 修改字段时从 `field_definitions_args.yaml` 开始。
4. 修改规则时从 `enhanced_rules_args.yaml` 开始。
5. 用 `/fields/retrieve` 和 `/client_search_query_parse_no_encipher` 做闭环验证。
6. 开发前优先修复测试路径迁移问题，否则全量测试无法作为回归保障。

# 09-批量问题评估系统使用说明

更新时间：2026-05-17

## 目标

用于把一批自然语言客户搜索问题批量提交给当前解析接口，自动保存解析结果、耗时、层级分布、失败样本和 Markdown 报告。

它和 `change_set.yaml` 迭代流水线共用同一套 evaluator，但入口更轻：适合直接评估已有问题清单。

## 启动服务

先启动当前 FastAPI 服务，确保下面接口可访问：

```bash
python -m src.main.python.main
```

默认评估地址是：

```text
http://localhost:8000/api/v1/client_search_query_parse_no_encipher
```

## 直接评估问题清单

支持 `.txt`、`.md`、`.jsonl`、`.csv`。

```bash
python -m src.main.python.tools.iteration_pipeline batch-eval \
  --input src/main/python/docs/测试集.txt \
  --base-url http://localhost:8000 \
  --concurrency 4
```

运行时会打印进度日志：

```text
loaded 100 cases from /path/questions.jsonl (graded=100, ungraded=0, concurrency=8)
[progress] 10/100 (10.0%) errors=0 avg_latency_ms=235.7 last=case_010 level=2 ok
[progress] 20/100 (20.0%) errors=0 avg_latency_ms=241.4 last=case_020 level=4 ok
```

默认输出目录：

```text
src/main/python/docs/evaluations/YYYYMMDD_HHMMSS/
```

生成文件：

- `batch_eval_result.json`：完整评估结果。
- `batch_failed_cases.jsonl`：有标准答案时，保存未完全匹配的样本。
- `batch_report.md`：汇总报告。
- `batch_eval_result.xlsx`：Excel 汇总，包含 `summary`、`cases`、`failed_cases` 三个 sheet。

## 输入格式

### 文本或 Markdown

每行一个问题。空行、标题行、Markdown 表格分隔行会被跳过。

```text
高温客户
找一下北京客户
1. 最近30天生日的客户
- 保单号包含ABC的客户
```

这种格式没有标准答案，只统计接口错误、解析层级和耗时。

### JSONL

推荐长期使用 JSONL。补充 `expected` 后会自动计算准确率。

```jsonl
{"id":"case_001","query":"高温客户","expected":{"query_logic":"AND","conditions":[{"field":"clientTemperature","operator":"MATCH","value":"高温"}]},"tags":["positive","l2"]}
{"id":"case_002","query":"今天温度高的客户","expected":{"query_logic":"AND","conditions":[]},"tags":["negative"]}
```

### CSV

CSV 至少包含 `query` 或 `question` 列，也支持中文列名 `问题`。

```csv
id,query,expected,tags
case_001,高温客户,"{""query_logic"":""AND"",""conditions"":[{""field"":""clientTemperature"",""operator"":""MATCH"",""value"":""高温""}]}",positive
```

## 常用参数

```bash
python -m src.main.python.tools.iteration_pipeline batch-eval \
  --input questions.jsonl \
  --output-dir src/main/python/docs/evaluations/manual_run \
  --base-url http://localhost:8000 \
  --timeout 30 \
  --concurrency 8 \
  --limit 100 \
  --progress-interval 5
```

进度相关参数：

- `--progress-interval N`：每完成 N 条输出一次进度，默认 10。
- `--progress-interval 1`：每条样本都输出一行，适合排查慢请求。
- `--no-progress`：只输出最终进度行和 summary。

## 生成候选标准答案

可以基于当前解析接口和现有配置，为未标注问题集生成候选 `expected`：

```bash
python -m src.main.python.tools.iteration_pipeline generate-expected \
  --input questions.jsonl \
  --output src/main/python/docs/evaluations/manual_run/candidate_expected.jsonl \
  --base-url http://localhost:8000 \
  --timeout 30 \
  --concurrency 8 \
  --progress-interval 5
```

同时会生成同名 Excel：

```text
src/main/python/docs/evaluations/manual_run/candidate_expected.xlsx
```

Excel 中包含 `expected_candidates` sheet，字段包括问题、候选 expected、标注状态、来源、是否需要复核等。
其中 `robot_text` 列为中文意图摘要，会优先使用解析返回的中文意图；如果没有，则根据 `expected.conditions` 和 `intent_summary_labels_args.yaml` 生成。

输出示例：

```jsonl
{"id":"case_001","query":"高温客户","expected":{"query_logic":"AND","conditions":[{"field":"clientTemperature","operator":"MATCH","value":"高温"}]},"label_status":"candidate","label_meta":{"source":"current_parse_api","needs_review":true,"matched_level":2}}
```

注意：这一步生成的是“候选标准答案”，不是天然正确的金标。建议人工抽查或修正后，再把它作为正式评估集使用。

默认情况下，解析为空的样本不会自动写入 `expected.conditions: []`，而是标记为：

```json
{"label_status":"manual_required","label_meta":{"reason":"empty_parse_result"}}
```

这样可以避免把系统当前没解析出来的问题误当成“应该为空”的负样本。如果确实要把空解析也生成负样本，可以加：

```bash
--include-empty-expected
```

也可以不经过 HTTP 端口，只基于配置中人工维护的示例生成候选标准答案：

```bash
python -m src.main.python.tools.iteration_pipeline generate-expected-from-config \
  --input questions.jsonl \
  --output src/main/python/docs/evaluations/manual_run/config_candidate_expected.jsonl \
  --timeout 30 \
  --concurrency 8 \
  --progress-interval 5
```

同时会生成：

```text
src/main/python/docs/evaluations/manual_run/config_candidate_expected.xlsx
```

默认模式只做 `field_definitions_args.yaml` 中 `examples` / `negative_examples` 的精确匹配：

- 命中 `examples[].query`：使用对应 `examples[].output` 生成 `expected`。
- 命中 `negative_examples[].query`：生成空条件负样本 `expected.conditions: []`。
- 未命中：标记 `manual_required`，不自动猜标准答案。

这种模式不会调用端口，也不会使用 L2 规则输出，因此更适合生成高可信候选金标。

如果确实要使用当前本地解析工作流生成低可信候选，可以显式加：

```bash
--use-parser-rules
```

`--use-parser-rules` 会在本地进程中加载 QueryRouter，并使用这些配置/规则：

- `enhanced_rules_args.yaml`：L2 正则、模板、组合规则。
- `field_definitions_args.yaml`：字段定义、examples、negative examples、字段校验。
- `field_enums_args.yaml` 和 `*_enums_args.yaml`：标准枚举值。
- `value_mappings_args.yaml`：口语值到标准值归一化。
- `intent_summary_labels_args.yaml`：摘要和不支持字段过滤逻辑。
- `dev_client_search_args.yaml`：当前环境开关、路径和基础解析约束。

如果还希望对规则无法覆盖的问题走 L4/LLM 候选标注，可以同时加：

```bash
--use-parser-rules --allow-l4
```

但 `--use-parser-rules` / `--allow-l4` 生成的候选更需要人工复核，因为 L2 规则和 L4 推理本身也是被评估对象。用于正式评估前，建议保留 `label_meta.needs_review: true`，抽查或修正后再沉淀为 gold 数据集。

## 多候选标注 Review

如果直接生成标准答案覆盖率太低，可以先生成多路候选，交给人工 review：

```bash
python -m src.main.python.tools.iteration_pipeline label-candidates \
  --input questions.jsonl \
  --output src/main/python/docs/evaluations/manual_run/label_candidates.xlsx \
  --write-jsonl
```

当前第一版不调用端口，也不使用 L2 最终输出，候选来源包括：

- `config_example`：`field_definitions_args.yaml` 中 examples / negative_examples 精确命中。
- `enum_overlap`：问题文本直接包含字段枚举值。
- `value_mapping`：问题文本命中 `value_mappings_args.yaml` 中的口语映射。

Excel 里会展示：

- `agreement_status`：`auto_approved`、`single_candidate_review`、`conflict_review`、`manual_required`。
- `candidate_1_expected` / `candidate_2_expected`：候选结构化答案。
- `candidate_1_evidence` / `candidate_2_evidence`：候选证据来源。
- `final_expected`：自动批准时填入；否则留给人工确认。
- `review_comment`：人工复核备注。

建议优先筛选：

```text
agreement_status != auto_approved
```

人工补完 `final_expected` 后，再导出成正式 gold JSONL 进行评估。

## 指标说明

- `graded_coverage_rate`：带 `expected` 标准答案的样本占比。
- `api_success_rate`：接口调用成功比例；不代表解析正确。
- `condition_non_empty_rate`：解析结果中 `conditions` 非空的比例；适合无标注问题集做覆盖观察。
- `known_level_rate`：返回了 `matched_level` 的样本比例。
- `total_accuracy`：总准确率；当前口径为已标注样本中完全匹配的比例。
- `exact_match_rate`：`query_logic` 和全部 `conditions` 完全匹配。
- `field_match_rate`：期望字段是否都被召回。
- `operator_match_rate`：期望字段和操作符是否都匹配。
- `empty_rate`：正样本解析为空的比例。
- `false_positive_rate`：负样本被错误解析出条件的比例。
- `avg_latency_ms` / `p95_latency_ms`：接口响应耗时。
- `level_distribution`：L1/L2/L3/L4 等解析层级分布。

如果输入文件没有 `expected`，`total_accuracy`、`exact_match_rate`、`field_match_rate`、`operator_match_rate` 会显示为 `N/A`。这是因为没有标准答案时无法判断模型解析是否正确，只能统计覆盖率、层级、耗时和错误。

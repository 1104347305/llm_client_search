# 08-自动化迭代流水线方案

更新时间：2026-05-13

## 目标

把“新增查询字段、维护 RAG 知识、维护 L2 规则模板、生成测试集、测试效果评估、按评估结果迭代优化”做成一条可重复运行、可追溯、可回滚的本地自动化流水线。

建议先做成本地 CLI，不急着上平台：

```bash
python -m src.main.python.tools.iteration_pipeline run --change-set docs/iterations/20260513_xxx/change_set.yaml
```

原因是当前项目的核心资产已经集中在 YAML、pytest、接口和 docs 中，本地 CLI 成本低，容易和现有开发习惯衔接。等流程稳定后，再接入 CI 或做 Web 管理页。

## 总体方案

流水线按一次“变更包”运行。每次要新增或优化的字段、枚举、规则、测试问题和预期结果，都记录在一个 `change_set.yaml` 中。

```text
变更需求
  -> change_set.yaml
  -> 配置生成/修改建议
  -> 配置静态校验
  -> 热更新与 RAG 重建
  -> 测试集生成
  -> 批量解析评估
  -> 失败样本归因
  -> 迭代建议
  -> docs/iterations/YYYYMMDD_xxx/ 归档
```

一次运行结束后，不只留下代码和配置，还要留下完整证据：输入是什么、改了什么、跑了哪些测试、效果如何、失败样本是什么、下一轮该改哪里。

## 推荐目录结构

```text
src/main/python/
  tools/
    iteration_pipeline/
      __init__.py
      cli.py
      change_set.py
      config_lint.py
      testset_generator.py
      evaluator.py

docs/
  iterations/
    20260513_customer_temperature/

      generated_testset.jsonl

```

如果不想马上新增 `tools/`，也可以先放在 `src/main/python/tests/automation/`，但长期建议放到 `tools/iteration_pipeline/`，避免和普通单元测试混在一起。

## Change Set 设计

`change_set.yaml` 是整条流水线的入口，建议保持人工可读。

```yaml
id: 20260513_customer_temperature
title: 客户温度字段优化
owner: mickey
reason: 支持高温、中温、低温客户查询

fields:
  - id: client_temperature_match
    field: clientTemperature
    operator: MATCH
    value_type: enum
    retrieval_text: 客户温度 高温客户 中温客户 低温客户
    enum: clientTemperature
    unit: null
    notes: 客户经营温度标签
    examples:
      - query: 高温客户
        value: 高温
      - query: 找一下中温客户
        value: 中温
    negative_examples:
      - query: 今天温度高的客户

enums:
  clientTemperature:
    values:
      - 高温
      - 中温
      - 低温
    aliases:
      高价值: 高温

l2_rules:
  - name: 客户温度匹配
    field: clientTemperature
    operator: MATCH
    value_type: capture
    enum_ref: clientTemperature
    patterns_template:
      - "{enum}客户"
      - "客户温度是{enum}"

test_requirements:
  min_positive_cases: 10
  min_negative_cases: 5
  include_logic_cases: true
  include_boundary_cases: true

acceptance:
  min_overall_accuracy: 0.95
  min_exact_match_rate: 0.95
  max_empty_rate: 0.02
  max_avg_latency_ms: 3000
```

第一版可以只让流水线读取它并生成报告，不自动改 YAML。等校验稳定后，再开启自动写入配置。

## 流水线阶段

### 1. 配置生成与合并

输入：`change_set.yaml`

输出：

- `field_definitions_args.yaml` 增量建议
- `field_enums_args.yaml` 或独立枚举文件增量建议
- `value_mappings_args.yaml` 增量建议
- `enhanced_rules_args.yaml` 增量建议
- `config_diff.patch`

建议策略：

- 默认生成 patch，由人 review 后应用。
- 对枚举、value mapping 这类低风险配置，可以支持 `--apply` 自动写入。
- 对 L2 正则规则，第一阶段只生成建议，因为误召回风险更高。

### 2. 静态校验

在运行服务前先做本地校验，尽早发现低级错误。

校验项：

- YAML 能否解析。
- intent `id` 是否唯一。
- `field` + `operator` 是否符合 `models/schemas.py`。
- `operator` 是否在 `Operator` 枚举内。
- `enum_ref` 是否存在。
- `value_mappings` 的目标值是否在枚举中。
- L2 `patterns` 是否能编译为合法正则。
- `patterns_template` 是否只引用已知变量，例如 `{enum}`、`{multi_enum}`。
- 新增字段是否出现在测试集和验收条件中。

建议命令：

```bash
python -m src.main.python.tools.iteration_pipeline lint --change-set docs/iterations/20260513_xxx/change_set.yaml
```

### 3. 热更新与 RAG 重建

服务启动后，流水线调用现有接口：

```http
POST /api/v1/config/reload/files
POST /api/v1/fields/reindex
POST /api/v1/fields/retrieve
```

推荐规则：

- 只改 L2、枚举、value mapping：先 reload，不强制 reindex。
- 改 `field_definitions_args.yaml` 的 `retrieval_text`、`examples`、`notes`：需要 reindex。
- 每次 reindex 后跑字段召回检查，记录 top_k 结果。

### 4. 测试集生成

测试集不要只靠手写，建议三层来源合并：

1. `change_set.yaml` 中的 examples 和 negative_examples。
2. 模板扩展：根据 operator、enum、字段别名生成固定表达。
3. LLM 辅助生成：只生成候选问题和预期结构，进入人工 review 后再转正。

输出文件建议使用 JSONL，方便 diff 和逐条复跑：

```json
{"id":"ct_001","query":"高温客户","expected":{"query_logic":"AND","conditions":[{"field":"clientTemperature","operator":"MATCH","value":"高温"}]},"tags":["positive","l2"]}
{"id":"ct_002","query":"今天温度高的客户","expected":{"conditions":[]},"tags":["negative"]}
```

测试集分层：

- `positive`：应该解析成功。
- `negative`：不能误召回。
- `boundary`：日期、年龄、枚举别名、空值等边界。
- `logic`：AND / OR / NOT / 组合查询。
- `regression`：历史失败样本，必须长期保留。

### 5. 批量评估

现有 `tests/batch_test.py` 已经能调用解析接口并导出 Excel，可以改造成 evaluator 的基础版本。

建议评估指标：

| 指标 | 说明 |
| --- | --- |
| overall_accuracy | 整体准确率；当前口径等同于整条样本完全匹配比例 |
| exact_match_rate | conditions、operator、value、query_logic 完全匹配比例 |
| field_match_rate | 字段命中比例 |
| operator_match_rate | 操作符命中比例 |
| value_match_rate | 值命中比例 |
| false_positive_rate | negative 样本被错误解析比例 |
| empty_rate | positive 样本解析为空比例 |
| level_distribution | L1/L2/L4 命中分布 |
| avg_latency_ms | 平均耗时 |
| p95_latency_ms | P95 耗时 |

第一版建议以 JSON 为主，Excel 为辅：

```text
eval_result.json
failed_cases.jsonl
report.md
```

### 6. 失败样本归因

每条失败样本至少归因到一个类别：

| 类别 | 常见处理 |
| --- | --- |
| field_not_recalled | 补 `retrieval_text`、`examples`、`notes`，然后 reindex |
| enum_not_normalized | 补枚举或 `value_mappings_args.yaml` |
| l2_miss | 补 `enhanced_rules_args.yaml` |
| l2_false_positive | 收紧正则上下文或加 negative case |
| llm_wrong_operator | 补 intent examples 或 prompt 约束 |
| condition_sanitized | 检查字段合法性和后处理过滤 |
| logic_wrong | 补组合查询样本和逻辑词测试 |

流水线可以先自动做粗归因：

- `/fields/retrieve` 没召回目标字段：`field_not_recalled`
- 召回字段对了但最终为空：`condition_sanitized` 或 `llm_parse_failed`
- L2 命中错误字段：`l2_false_positive`
- 枚举值不在标准值列表：`enum_not_normalized`

## 推荐实施路线

### 第一阶段：可记录

目标：不自动改配置，只把流程跑通并沉淀文档。

交付：

- `docs/iterations/` 目录。
- `change_set.yaml` 模板。
- 从 JSONL 测试集调用接口并生成 `eval_result.json`、`failed_cases.jsonl`、`report.md`。
- README 增加自动化流程入口。

### 第二阶段：可校验

目标：新增配置静态校验和验收阈值。

交付：

- YAML schema 校验。
- L2 正则编译校验。
- enum/value mapping 一致性校验。
- `acceptance` 阈值判断。
- 失败时返回非 0 exit code，方便接 CI。

### 第三阶段：可生成

目标：根据 change set 生成测试集和配置 patch。

交付：

- 自动生成 positive / negative / boundary / logic 测试候选。
- 自动生成配置 patch。
- 人工 review 后再 apply。

### 第四阶段：可优化

目标：基于失败样本生成下一轮修改建议。

交付：

- 失败归因。
- 对 `retrieval_text`、examples、L2 patterns、value mappings 的建议。
- 历史失败样本自动沉淀到 regression 集。

## 建议优先做的最小版本

最小版本不要追求“一键全自动改好”，先追求“每次迭代都有证据、有指标、有归档”。

优先级：

1. 建立 `docs/iterations/` 和 `change_set.yaml` 模板。
2. 把 `batch_test.py` 改造成读取 JSONL 预期结果并输出评估指标。
3. 生成 `report.md` 到本地 docs。
4. 增加配置静态校验。
5. 再做测试集和配置 patch 自动生成。

这样可以很快解决当前最痛的问题：新增字段和规则以后，效果到底有没有变好、哪些样本失败、为什么失败、下一轮要改什么，都能在本地 docs 中追踪。

## 当前已落地的最小实现

代码位置：

```text
src/main/python/tools/iteration_pipeline/
```

模板位置：

```text
src/main/python/docs/iterations/change_set.template.yaml
```

### 1. 创建一次迭代目录

```bash
mkdir -p src/main/python/docs/iterations/20260514_customer_temperature
python -m src.main.python.tools.iteration_pipeline init \
  --output src/main/python/docs/iterations/20260514_customer_temperature/change_set.yaml
```

然后编辑生成的 `change_set.yaml`，填入本轮要新增的字段、枚举、value mapping、L2 规则和验收阈值。

### 2. 静态校验 change set

```bash
python -m src.main.python.tools.iteration_pipeline lint \
  --change-set src/main/python/docs/iterations/20260514_customer_temperature/change_set.yaml
```

当前校验内容：

- 必填字段是否存在。
- intent id 是否和现有 `field_definitions_args.yaml` 冲突。
- operator 是否属于 `models/schemas.py` 中的 `Operator`。
- enum_ref 是否存在。
- value_mappings 是否是合法对象。
- L2 正则是否能编译。
- test_cases 中的 expected operator / query_logic 是否合法。

### 3. 生成 JSONL 测试集

```bash
python -m src.main.python.tools.iteration_pipeline generate-testset \
  --change-set src/main/python/docs/iterations/20260514_customer_temperature/change_set.yaml
```

默认会写入 `change_set.yaml` 中的 `testset_path`，例如：

```text
src/main/python/docs/iterations/20260514_customer_temperature/generated_testset.jsonl
```

测试来源包括：

- `test_cases` 中手工写的用例。
- `fields[].examples` 自动转换出的 positive 用例。
- `fields[].negative_examples` 自动转换出的 negative 用例。

### 4. 启动服务并执行评估

先启动服务：

```bash
ENV=dev python -m src.main.python.main
```

再运行评估：

```bash
python -m src.main.python.tools.iteration_pipeline run \
  --change-set src/main/python/docs/iterations/20260514_customer_temperature/change_set.yaml \
  --base-url http://localhost:8000
```

执行后会在同一个迭代目录生成：

```text
eval_result.json
failed_cases.jsonl
report.md
```

### 5. 看报告并迭代

优先看：

```text
report.md
failed_cases.jsonl
```

常见处理：

- `empty_result_or_field_not_recalled`：补 `retrieval_text` / `examples` / `notes`，然后重建字段索引。
- `false_positive`：收紧 L2 正则上下文，或增加 negative case。
- `condition_missing_or_value_wrong`：检查枚举归一化、operator、value_type。
- `logic_wrong`：补组合查询样本和逻辑词规则。

## 新增一个字段解析

适用场景：用户想支持一个新的客户字段，比如“客户等级”“保单状态”“家庭成员职业”等。

步骤：

1. 在 `config/field_definitions_args.yaml` 增加字段 intent。
2. 如果字段有枚举，在 `field_enums_args.yaml` 或独立 `*_enums_args.yaml` 增加枚举。
3. 如果用户会使用口语表达，在 `value_mappings_args.yaml` 增加归一化。
4. 如果希望 L2 直接命中，在 `enhanced_rules_args.yaml` 增加规则。
5. 调 `/api/v1/config/reload/files` 热更新配置。
6. 如果字段需要 ES RAG，调 `/api/v1/fields/reindex`。
7. 调 `/api/v1/fields/retrieve` 检查字段召回。
8. 调 `/api/v1/client_search_query_parse_no_encipher` 检查最终 conditions。
9. 增加或修复测试。

热更新示例：

```bash
curl -X POST http://localhost:8000/api/v1/config/reload/files \
  -H "Content-Type: application/json" \
  -d '{
    "files": ["field_definitions", "enhanced_rules", "field_enums", "value_mappings"],
    "force_reindex_fields": false,
    "wait": true
  }'
```

重建字段索引：

```bash
curl -X POST http://localhost:8000/api/v1/fields/reindex
```

## 新增一个 L2 规则

适用场景：某类表达比较固定，希望不用 LLM 就能解析。

基础规则示例：

```yaml
- name: "客户年龄大于等于"
  field: "clientAge"
  operator: "GTE"
  value_type: "capture"
  patterns:
    - "(\\d+)岁以上"
```

枚举规则示例：

```yaml
- name: "持有产品简称"
  field: "polNoInfo.plancodeinfo.abbrname"
  operator: "CONTAINS"
  value_type: "capture"
  enum_ref: "polNoInfo.plancodeinfo.abbrname"
  patterns_template:
    - "买过{enum}"
    - "购买了{enum}"
```

开发注意：

- 正则要尽量加上下文，减少误召回。
- 多枚举表达使用 `{multi_enum}`。
- 否定表达优先用 `NOT_CONTAINS` 或 `NOT_EXISTS`。
- 涉及年龄、日期、保单、家庭成员字段时，要检查已有边界测试。
- 新增后先跑针对性测试，再跑全量测试。

## 新增枚举值

适用场景：新增产品名、职业名、客户状态等。

步骤：

1. 找到对应枚举文件。
2. 增加标准枚举值。
3. 在 `value_mappings_args.yaml` 增加别名到标准值的映射。
4. 热更新枚举和 value mappings。
5. 如果影响 L4 RAG，重建字段索引。
6. 用 `/fields/retrieve` 检查能否召回。

普通枚举一般在：

```text
config/field_enums_args.yaml
```

大枚举一般在：

```text
config/*_enums_args.yaml
```

## 调试 L4 不识别的问题

先不要急着改 prompt。建议按下面顺序查：

1. 调 `/api/v1/fields/retrieve`，看目标字段是否被召回。
2. 看 `prompt_section` 是否包含目标字段的 `field`、`operator`、`enum`、`examples`。
3. 如果没有召回，补 `field_definitions_args.yaml` 的 `retrieval_text`、`examples`、`notes`。
4. 如果枚举没命中，检查对应 `*_enums_args.yaml` 和 `value_mappings_args.yaml`。
5. 如果 L2 候选应当召回但没有，检查 `enhanced_rules_args.yaml` 的规则上下文。
6. 再调 `/client_search_query_parse_no_encipher`，看 `matched_level` 和 `matched_patterns`。

字段召回示例：

```bash
curl -X POST http://localhost:8000/api/v1/fields/retrieve \
  -H "Content-Type: application/json" \
  -d '{"query":"买过重疾险的客户","top_k":8}'
```

## 调试解析结果为空

常见原因：

- L1/L2 没命中。
- L4 RAG 没召回任何字段，主动跳过 LLM。
- L4 输出了非法字段，被 `_validate_conditions()` 清空。
- 枚举值未归一，导致结果被后处理丢弃。
- 查询涉及不支持的角色字段，被 `_sanitize_unsupported_conditions()` 删除。

建议查看：

```text
matched_level
matched_patterns
rewritten_query
conditions
intent_summary
logs/app.log
```

## 恢复或新增搜索接口

当前代码已有：

```text
services/search_service.py
services/search_api_client.py
```

但 router 当前没有暴露搜索接口。如果要恢复：

1. 在 `api/client_search_query_parse_post.py` 增加 route。
2. 自然语言搜索调用 `SearchService.natural_language_search()`。
3. 结构化搜索调用 `SearchService.structured_search()`。
4. 确认下游 `SEARCH_API_BASE_URL` 可用。
5. 补请求日志。
6. 补接口测试和下游 HTTP mock。

## 提交前检查清单

- 字段是否在 `field_definitions_args.yaml` 有 intent。
- 字段是否在下游搜索 API 支持。
- operator 是否是 `models/schemas.py` 中定义的枚举。
- 枚举值是否是标准值。
- 口语值是否在 `value_mappings_args.yaml` 归一化。
- L2 规则是否会误伤相似语句。
- L4 RAG 是否能召回目标字段。
- 是否需要 `/fields/reindex`。
- 是否补充测试。
- 是否避免提交真实密钥。

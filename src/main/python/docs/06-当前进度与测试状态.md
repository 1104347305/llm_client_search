## 变更记录

### 2026-05-24：单值互斥字段多值查询逻辑修正

- 背景：客户姓名、客户温度、客户价值等字段对单个客户而言不可能同时存在多个取值。用户输入 `王进李四`、多个客户温度或多个客户价值候选时，最终语义应是任选其一，而不是同时满足。
- 实现：在 `field_mapping_args.yaml` 中新增 `single_value_or_fields` 配置，默认包含 `searchClientName`、`clientTemperature`、`newValueLabel`；`models/field_mapping.py` 暴露 `get_single_value_or_fields()`。
- 后处理：`QueryRouter` 在最终组装 `ParsedQuery` 前检查条件；若最终条件只涉及一个配置字段，且该字段解析出多个值，则将外层 `query_logic` 修正为 `OR`。
- 口径：该规则只改最终逻辑，不改字段识别、值抽取和 operator；多字段组合仍保持原有 `AND` 语义，未配置字段的多值 `CONTAINS` 不受影响。
- 回归：更新 `test_query_logic_for_contains_list.py`，覆盖配置字段多值转 `OR` 和非配置字段多值保持 `AND`。执行 `pytest -q src/main/python/tests/test_query_logic_for_contains_list.py`，结果为 `2 passed`。

### 2026-05-15：L2 投保险种大小写无关匹配

- 背景：投保险种名称/简称中存在英文字母，例如 `e生保`、`平安e生保医疗保险`。用户可能输入 `E生保`、`平安E生保医疗保险`，期望 L2 按同一标准枚举识别。
- 实现：在 `enhanced_rules_args.yaml` 中为投保险种名称、未持有投保险种名称、投保险种简称、未持有投保险种简称规则增加 `ignore_case: true`；同时为 `投保险种简称+投保险种类型`、`投保险种简称+保单状态`、`投保险种名称+保单状态` 组合规则增加 `ignore_case: true`。
- 标准化：在 `value_mappings_args.yaml` 中补充 `E生保 -> e生保`、`平安E生保医疗保险 -> 平安e生保医疗保险`，确保最终输出值仍是标准枚举值，而不是用户输入的大小写变体。
- 口径：本次没有全量开启 L2 大小写无关匹配，只限定在投保险种名称/简称及相关组合规则，避免放宽身份证、案件号、客户号、裸值弱匹配等规则的召回边界。
- 预期输出：`买了E生保产品，且保单状态为缴费有效的客户` 输出 `polNoInfo.plancodeinfo.abbrname MATCH e生保` + `polNoInfo.polStatus MATCH 交费有效`。
- 回归：在 `test_l2_policy_plan_composites.py` 中新增 `test_l2_matches_policy_plan_case_insensitive_variants`，并将相关组合规则用例对齐到当前枚举中的 `e生保`。执行 `python -m pytest src/main/python/tests/test_l2_policy_plan_composites.py -q`，结果为 `16 passed, 1 warning`。
- 热更新：本地 shell 调用 `POST /api/v1/config/reload/files` 时无法连接 `localhost:8000`，未完成运行时热加载；服务可用时需刷新 `enhanced_rules` 与 `value_mappings`。

### 2026-05-14：L2 客户对象尾缀抽取为公共 pattern

- 背景：L2 规则中多处复用 `(?:的客户|客户|有哪些客户|有哪些人|名单|的人|人)?` 作为查询对象尾缀，维护成本较高。
- 实现：在 `enhanced_rules_args.yaml` 的 `pattern_vars` 中新增 `CUSTOMER_SUFFIX`，并将客户价值相关的完整等价尾缀替换为 `{CUSTOMER_SUFFIX}`。
- 范围：本次只替换完全等价的完整尾缀，短版 `(?:的客户|客户)?` 等变体保持不变，避免扩大匹配口径。
- 回归：新增 `test_l2_customer_suffix_pattern_var.py`，验证变量展开无残留，并确认 `高价值名单` 仍可命中 `newValueLabel CONTAINS [A1, A2, A3, A4, B, C]`。

### 2026-05-14：e生保生效客户解析口径修正

- 问题：`e生保生效客户` 容易被泛化解析为“存在保单生效日”，即 `polNoInfo.poleffdate EXISTS`。
- 口径：这里的“生效客户/生效中客户”表示保单状态有效，不表示有保单生效日。
- 实现：将口语表达 `生效`、`生效中` 合并到 L2 规则 `保单状态-有效汇总`，并让“投保险种简称/名称 + 保单状态”组合规则引用该汇总规则。
- 预期输出：`polNoInfo.plancodeinfo.abbrname MATCH e生保` + `polNoInfo.polStatus CONTAINS [交费有效, 自垫交清, 交清, 减额交清, 免交, 自垫有效]`。
- 回归：新增 `test_l2_maps_eshenbao_effective_customer_to_policy_status_not_effective_date`，明确断言不输出 `polNoInfo.poleffdate`。

### 2026-05-14：理赔案件号 MC 格式 L2 模板补充

- 问题：`理赔案件号-MC格式` 已有模板中的 `\d{4, 14}` 量词含空格，导致 MC 号不能按预期匹配；同时缺少类似手机号的开头、尾号、反向表达模板。

- 预期输出：统一输出 `polNoInfo.claimdatainfo.claimno MATCH <提取值>`。
- 回归：新增 `test_l2_claim_case_no.py`，覆盖全量 MC 号、裸值、前缀和尾号表达。

### 2026-05-14：日期类 L2 相对时间模板补充

- 背景：除生日、应缴日外，保单周年日、证件到期、缴费期满、保单到期、承保、理赔、保单生效、客户添加等日期字段对 `本周/下周/下下周/上周/今天/明天/后天/未来N天` 的覆盖不均。

## 主要失败原因

### 1. 测试仍引用旧路径

多个测试还在引用迁移前的模块路径：

```python
from core.level2_enhanced_matcher import Level2EnhancedMatcher
from core.query_router import QueryRouter
from core.field_registry import FieldRegistry
from services.search_service import SearchService
```

但当前源码路径已经迁移为：

```python
from src.main.python.steps.level2_enhanced_matcher import Level2EnhancedMatcher
from src.main.python.steps.query_router import QueryRouter
from src.main.python.steps.field_registry import FieldRegistry
from src.main.python.services.search_service import SearchService
```

### 2. 部分测试文件存在语法错误

例如：

```text
src/main/python/tests/test_level1.py
src/main/python/tests/test_logic_operators.py
```

错误形态：

```python
from sys.path.insert(...)
```

这不是合法 Python 语法，应改为：

```python
import sys
sys.path.insert(0, "...")
```

或直接移除旧路径注入，统一使用新包路径。

### 3. 部分测试引用不存在模块

例如：

```python
from core.level2_template_matcher import Level2TemplateMatcher
```

当前项目中没有对应模块，需要确认是历史遗留测试还是模块迁移遗漏。

## 当前风险点

`git status` 显示大量旧根目录文件删除、新 `src/main/python/` 文件新增/移动。后续合并前需要确认这是预期迁移，而不是误删。

`SearchService` 与 `SearchAPIClient` 存在，但当前 FastAPI router 没有暴露自然语言搜索或结构化搜索 route。

### 配置文件包含明文密钥

当前环境 YAML 中存在 `LLM_API_KEY`。建议迁移到环境变量或密钥管理系统。

```python

```

pytest 输出 FastAPI deprecation warning。后续可以迁移到 lifespan。
